Part 1

Variational Autoencoder (VAE)

In Part 1, I implemented a basic autoencoder and a VAE, both trained on the MNIST dataset (60,000 training images, 10,000 test images of handwritten digits). Both models use a simple fully connected network: the encoder compresses the input image (784 pixels) down to a small latent vector, and the decoder tries to reconstruct the image from that vector. The autoencoder uses MSE loss to measure reconstruction error, while the VAE also adds a KL divergence term to push the latent space toward a standard normal distribution. I used Adam with a learning rate of 0.001 and a batch size of 128 for both.

Task 1 — Basic Autoencoder

I trained three versions of the autoencoder with different latent sizes: 2, 8, and 32 dimensions, each for 20 epochs. Figure 1 shows how the training loss changed over time for all three.

The final losses were about 0.035 for latent=2, 0.0135 for latent=8, and under 0.009 for latent=32. This makes sense — the more dimensions you give the model, the more information it can store and the better it can reconstruct the original image.

Reconstruction Quality

Discussion: The bigger the latent dimension, the better the reconstructions get. With only 2 dimensions, the model has to squish an entire digit into just 2 numbers, so a lot of detail gets lost and the output looks blurry. With 32 dimensions, there's enough space to store all the important features and the reconstructions look nearly perfect. The downside of having more dimensions is that random sampling becomes harder, which is what motivated me to try the VAE next.

Sampling from the Latent Space

Discussion: The autoencoder is great at compressing and reconstructing images it's seen before, but it's bad at generating new ones. The reason is that during training, the encoder can place each image's code anywhere in the latent space without any restrictions. So there are big empty gaps between the clusters of codes, and if you sample a random vector it usually lands in one of those gaps, giving you garbage output. This is exactly the problem that VAEs are designed to fix.

Task 2 — Variational Autoencoder

For the VAE I used a 2-dimensional latent space so I could visualize it directly. Instead of encoding each image to a single point, the VAE encoder outputs a mean (μ) and a variance (σ²) for each dimension. To get the actual latent vector, I use the reparameterization trick: z = μ + ε · σ where ε is sampled from N(0,1). This makes the whole thing differentiable so we can train it with backprop.

The loss function has two parts: binary cross-entropy to measure how well the image was reconstructed, plus a KL divergence term that penalizes the model whenever the learned distribution drifts away from N(0,1). This KL term is what forces the latent space to be smooth and continuous.

VAE vs. AE Sampling

Discussion: The VAE generates much more realistic-looking digits than the regular autoencoder. The KL divergence loss forces all the latent codes to stay close to N(0,1), so the space is filled in much more densely. When you sample a random vector, it's much more likely to land somewhere the decoder has seen before, so the output actually looks like a digit. The tradeoff is that VAE reconstructions tend to be a bit blurrier than the AE's because the BCE loss kind of averages things out, but for generation purposes the VAE is clearly better.

Latent Space Visualization

Discussion: The latent space visualizations show that the VAE has organized the digits in a pretty meaningful way. In the [−1, 1] grid you can see gradual transitions between different digit types — nearby points in the latent space decode to visually similar images. When we zoom out to [−5, 5], things start breaking down at the edges. This makes sense because N(0,1) puts very little probability mass out that far, so the model never really learned what those extreme points should look like. Based on this, the sweet spot for generating good images seems to be roughly in the [−2.5, 2.5] range.
Part 2

Deep Convolutional GAN

In Part 2, I implemented a DCGAN to generate Pokémon images. The dataset has 809 images stored in Google Drive, all resized to 64×64 RGB. The generator takes a random 100-dimensional noise vector and uses a series of transposed convolution layers to upsample it into a 64×64 image. The discriminator is basically a regular CNN that takes an image and outputs a single number between 0 and 1 — its confidence that the image is real. Both networks are initialized with small random weights as described in the original DCGAN paper, and both are trained with Adam using β₁ = 0.5.

Task 1 — DCGAN Implementation

The generator has 5 layers total: it starts from a 4×4 feature map and doubles the spatial size at each step until reaching 64×64. Each layer uses a transposed convolution, batch normalization, and ReLU, except the last layer which uses Tanh to produce outputs in the [−1, 1] range. The discriminator mirrors this with regular convolutions, batch normalization, and LeakyReLU (skipping batch norm on the first layer as the paper recommends). I saved a fixed noise vector at the start of training so I could consistently track how the generated images improve over epochs.

Task 2 — GAN Tuning and Evaluation

Loss Curves (50 Epochs, LeakyReLU)

Discussion: The training didn't go very well. The discriminator gets really good almost immediately (loss close to 0 within the first few epochs), which means it can easily tell the difference between real and fake images. Once that happens, the generator basically stops receiving useful feedback and its loss just keeps going up. The main reason for this is probably the small dataset — with only 809 real images, the discriminator can just memorize them all. To fix this you'd want to try things like data augmentation, adding noise to the discriminator inputs, or using a lower learning rate for the discriminator.

Image Quality Progression

LeakyReLU vs. Standard ReLU

Discussion: Both versions ended up with the same problem — the discriminator dominates quickly. However, with standard ReLU the discriminator collapsed a bit faster. This is likely because ReLU kills all negative activations, which can cause neurons to stop updating (the "dying ReLU" problem). With LeakyReLU, negative values still produce a small gradient, so more neurons stay active and the discriminator doesn't just short-circuit. In practice though, both activations failed to produce a stable training run on this dataset, so the difference was mostly visible in how fast the collapse happened rather than whether it happened.

Best setting — lr=0.0001, slope=0.1: This was the only combination where the discriminator loss didn't immediately go to zero. At 0.01, it still has a non-trivial loss, which means both networks are still updating. Everything else at lr=0.0002 or higher just immediately had the discriminator win.
Interesting outlier — lr=0.0004, slope=0.1: Here the generator actually had a lower loss (3.64) while the discriminator struggled (1.21). This is the opposite of what normally happens. It's probably just a lucky random initialization and wouldn't hold up over more epochs, but it shows that with a high enough learning rate the generator can sometimes temporarily overpower the discriminator.
Mode collapse: Looking at the image progression, the generator seems to be producing similar-looking blobs by the end of training rather than diverse Pokémon. This is a sign of mode collapse, where the generator finds a few outputs that fool the discriminator and just keeps making those. Again, the small dataset is the main culprit here.

Review

Code Review Notes

Overall the code is correct and meets all the requirements. I noticed one small bug in Part 2 and a couple things worth mentioning.

Bug in train_dcgan (Part 2): The function takes a parameter called label (a string like 'LeakyReLU') for use in the print statements. But inside the training loop, the same variable name gets reused for the PyTorch label tensor: label = torch.full((b,), real_label, ...). After that point, the print statement f'[{label}] Epoch ...' prints the whole tensor of ones instead of the string. You can see this in the notebook output. Training still works fine, but the logs look messy. The fix is simple — just rename the string parameter to something like run_name so the two variables don't conflict.
Part 1 — correctness: Everything looks right. The reparameterization trick is implemented correctly. The KL divergence formula matches the standard closed-form version used in the original VAE paper. The latent grid visualization correctly decodes a 2D grid of points, and using ToTensor() puts pixel values in [0,1] which is what BCE loss expects.
Part 2 — correctness: The DCGAN follows the original paper correctly: transposed convolutions with BatchNorm and ReLU in the generator, strided convolutions with BatchNorm and LeakyReLU in the discriminator (no BatchNorm on the first discriminator layer), Tanh at the end of the generator and Sigmoid at the end of the discriminator. The weight initialization uses N(0, 0.02) as specified. One thing that's done well is calling fake_imgs.detach() when updating the discriminator — this prevents gradients from flowing back into the generator during that step, which is important for stable training.
Summary

Conclusion

This project covered two different approaches to generative modeling. The VAE worked pretty well — it produced clear, recognizable digits and the 2D latent space visualization showed that it learned a smooth, organized representation of the data. The DCGAN was more challenging. The architecture was implemented correctly, but the small Pokémon dataset (only 809 images) made it really hard to get stable training — the discriminator kept overpowering the generator before it could learn much.
